<objective>
Implement the deterministic quality engine: `lint-requirements` (TOOL-04, TOOL-05) and
`verify` (TOOL-06). `lint-requirements` runs regex/word-set heuristics for grounding,
verifiability, atomicity, citation (FAIL-class per D-07) and ambiguity (WARN-class), plus
REQ-ID stability — a two-pass material-change detector (RESEARCH Pattern 5 + Pitfall 6:
pass 1 = existing ID with a changed statement; pass 2 = a NEW ID whose normalized statement
matches an OLD statement, catching silent renumbering) which is ALWAYS FAIL. `verify` folds
the lint result and adds the hardened grounding gate: open `source_trace.doc` and confirm
`source_trace.span` is a real >=12-char verbatim substring scoped to the cited section
(RESEARCH Pattern 3 + Pitfall 2 heading normalization), with `--cite-scope document` override
(REQUIREMENTS Open Decision #1). `verify` exits non-zero ONLY on a FAIL-class finding (D-08).

Purpose: this is the verification core (DESIGN §5/§6) — the determinism boundary in action:
the CLI proves the citation is REAL; whether it JUSTIFIES the statement stays ba-critic's
call (Phase 2). Heuristics are deterministic patterns, never ML/NLP.
Output: `lint.py`, `citation.py`, `lint-requirements` + `verify` commands, and their tests.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| requirements/source file paths -> filesystem | `--requirements`, `--config`, and `source_trace.doc` are untrusted paths |
| requirement statement text -> lint heuristics | untrusted text drives regex checks (ReDoS surface) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-1-01 | Tampering | `verify` source-doc path resolution | mitigate | `source_trace.doc` and `--requirements` resolved under repo root and gated by `is_within_root` before read; escaping path -> FAIL, never read outside root |
| T-1-09 | Tampering | citation file read escape | mitigate | Citation source path validated `is_within_root` (T-1-09 from RESEARCH Security Domain) before `read_text` |
| T-1-11 | Denial of Service | regex lint heuristics on hostile input | mitigate | Use simple linear-time patterns (`\b`-anchored word matches, fixed substring `in`), no catastrophic-backtracking regex; spans are plain substring `in` checks (no regex on the span) |
</threat_model>
