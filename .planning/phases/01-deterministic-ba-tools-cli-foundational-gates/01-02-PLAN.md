<objective>
Implement two of the four foundational gates: deterministic route resolution
(TOOL-02) and the byte-check gate (GATE-04 / CDX-04). `resolve-route` returns ONLY
the static `DEFAULT_ROUTES` mapping and raises `BaToolsError` (exit 2) for an unknown
operator — it never parses, guesses, or infers a route from free text (DESIGN §11
non-negotiable). `byte-check` fails the build (exit 2) when any eager-loaded doc is
>= 32,768 bytes (the Codex silent-truncation limit, CLAUDE.md CONFIRMED) and passes
when under, resolving every path relative to `--repo-root`.

Purpose: lock the two cheapest-to-verify gates so later operators consume a fixed
route table and a working size gate. Both are pure file/command-provable work (no
judgement) — squarely inside the determinism boundary.
Output: working `resolve-route` and `byte-check` subcommands with passing tests.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| CLI path args -> filesystem | `byte-check` path arguments and `--repo-root` are untrusted and resolve to real files |
| operator arg -> route table | `resolve-route` operator string is untrusted; must only key a static dict |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-1-01 | Tampering | `byte_check.run` path resolution | mitigate | Each path resolved as `(repo_root / raw).resolve()` and gated by `is_within_root`; escaping paths raise PATH_ESCAPE (no read outside root) |
| T-1-04 | Spoofing/Tampering | `resolve_route` free-text inference | mitigate | Route comes ONLY from the static `DEFAULT_ROUTES` dict key lookup; unknown operator -> exit 2; no parsing of free text (DESIGN §11) |
| T-1-07 | Information Disclosure | byte-check / resolve-route errors | accept | Failures are structured `{code, path/operator}`; no stack traces or secrets surface (inherited from 01-01 handler) |
</threat_model>
